## 1. 專案目標與定位

- **產品定位**:Splatoon 3 玩家用的武器查詢與輔助工具(非官方、非營利)。
- **目標用戶**:Splatoon 3 玩家。
- **優先級排序(由高到低)**:
  1. 對玩家的實用性與回訪價值
  2. SVG 美術與動態表現
  3. 開發速度
  4. 前端工程深度
- **功能主從關係**(訪談釐清):
  - **核心**:武器資料查詢(玩家真正回訪的理由)。
  - **附加趣味**:隨機武器決定器。
  - **視覺亮點**:11 分類 SVG 美術。

> 備註:純資料查詢的回訪屬「不定期、低頻」(想到才查),不是賽程那種「每日固定回來」。已確認本專案不納入賽程/排程類功能,接受此回訪強度。

## 2. 範圍 (Scope)

### 2.1 納入範圍
- 武器資料查詢(列表 + 詳情),靜態表格呈現。
- 隨機武器決定器,支援多重條件篩選(分類 / 副武器 / 特殊武器)。
- 11 個武器分類各一個代表武器的 SVG 美術,含最低限度動態。
- 多語系(ja-JP / zh-TW / en)。
- 手機優先 (mobile-first) 的響應式設計。

### 2.2 排除範圍(明確不做)
- 賽程 / 鮭魚跑排程追蹤。
- 使用率 / 勝率等統計資料。
- 互動式裝備效益計算機(輸入 AP 算結果)。
- 兩把武器 / 配裝的比較功能。
- 玩家個人對戰資料讀取(不碰 SplatNet 非公開 API、不做 token 生成)。

## 3. 功能規格

### 3.1 武器資料查詢(核心)

**列表頁**
- 以分類分組或可切換的清單呈現所有主武器。
- 每個項目顯示:武器名稱、所屬分類、副武器、特殊武器、自繪的分類 SVG 縮圖。
- 支援篩選(分類 / 副 / 特殊)與文字搜尋。
- `[opt-in 選用]` 列表與詳情頁可顯示官方武器圖示,**預設關閉**,僅在環境變數開啟時於執行時外部載入(界線見 §4.3.1)。

**詳情頁**
- 基本資訊:名稱、分類、副武器、特殊武器(副 / 特殊僅顯示名稱 + 一行簡述,不做獨立詳情頁)。
- 戰鬥數值:採「精簡核心子集」(見下方定義),以靜態表格列出。
- 裝備效益數值:各裝備效益的效果量(以靜態表格列出,不做互動計算)。
- 嵌入該武器所屬分類的 SVG 美術。

> `[已定案]` 戰鬥數值欄位 = **精簡核心子集**,固定呈現以下四類,其餘參數留待日後擴充:
> 1. **傷害**:可能多段(如直擊 / 範圍、近 / 遠),以「帶標籤的數值組」表示。
> 2. **射程**:採遊戲內 0–100 的相對射程值。
> 3. **墨水消耗**:每次發射消耗(占墨水槽 %)。
> 4. **連射 / 充能(節奏)**:依分類取最具代表性的節奏參數 — 射手類用連射間隔;蓄力類(チャージャー / スピナー)用滿充時間;其餘分類取該類的代表節奏值。
>
> `[已定案]` 副武器 / 特殊武器**不做獨立詳情頁**,僅在主武器詳情頁顯示其名稱與一行簡述(保持資訊聚焦在主武器,降低範圍與 schema 複雜度)。

### 3.2 隨機武器決定器(附加)

- 定位:純好玩,玩家每場開打前抽一把。
- 抽選範圍支援多重條件篩選:可限定分類、副武器、特殊武器、射程區間等條件後再抽。
- **多槽抽選**:一次抽選可由 1–8 個「抽選槽」組成,每槽各自獨立條件(分類 / 副 / 特殊 / 射程區間),按一次「全部抽選」同時揭曉(單把抽選 = 1 槽的特例)。
- 抽中結果顯示該武器資訊,並以其分類 SVG 作為視覺結果。
- 預設為無狀態(stateless),不需記錄歷史。

> `[已定案]` **不做**「排除剛抽過、避免連續重複」功能,維持無狀態純抽選(允許連抽到同一把),保持單純;可日後再加。
>
> `[補述 2026-06]` 多槽抽選新增「單次內去重」選項(同一次抽出的多把之間不重複)。此選項**不違反**上方定案:去重僅作用於單一次抽選的當下,跨次仍為無狀態(不記錄歷史、不排除上一次抽過的武器)。

### 3.3 SVG 美術(視覺亮點)

- 對 Splatoon 3 的 **11 個主武器分類**各繪製一個代表武器的 SVG。
- 動態程度:最低限度(待機微動 + hover 或抽中時的回饋動畫)。
- 用途:
  1. 嵌入武器詳情頁(顯示其分類圖)。
  2. 作為隨機決定器抽中後的視覺結果。
  3. (選用)獨立的分類美術一覽。

## 4. 資料來源與合規(關鍵)

> 立場聲明:以下為著作權一般原則下的風險評估,非法律意見。核心策略是「只用事實性數據、名稱走開放資料、美術全自繪、非營利、標明非官方」。

### 4.1 名稱與多語系字串 — 來源:splatoon3.ink(開放)
- splatoon3.ink 的 locale 資料提供 ja-JP / zh-TW / en 等語言字串,並明確開放他人用於自己的網站 / app。
- **必須遵守的條件**:
  - 在專案中標註資料來源 splatoon3.ink。
  - 做快取,抓取頻率不高於每小時一次。
  - 送請求時設定 User-Agent。
  - 使用該資料的產品須免費提供(可接受自願捐款)。

### 4.2 戰鬥數值與裝備效益數值 — 來源:Leanny 參數資料(風險低,有但書)
- 數值本身屬遊戲「事實性數據」,事實不受著作權保護,風險低。
- **但書**:
  - 不可原樣照抄 Leanny 的資料庫結構與編排(避免觸及「編輯著作權」)。需自行定義資料 schema、自建一份快照 JSON。
  - Leanny repo 未發現明確開源授權,不能主張獲得授權,僅以「事實性數據」立場使用。
- **自建 schema(對應 §3.1 精簡核心子集)**:每把主武器一筆,概念欄位如下;以扁平、語意命名的自有結構表達,**不沿用** Leanny 的 key 命名與巢狀編排。
  - `id`、`category`(11 分類之一)、`subWeaponId`、`specialWeaponId`
  - `coreStats`:節奏可變的數值陣列 `{ key, value, unit }`,key 限於 `damage` / `range` / `inkConsumption` / `cadence` 四類(`cadence` 視分類為連射間隔或滿充時間)
  - `gearEffects`:各裝備效益的效果量(靜態列出,不互動計算)
  - `gameVersion`:本筆數值對應的遊戲版本(快照標註)
  - 名稱(ja-JP / zh-TW / en)走 splatoon3.ink locale,以 id 對應,不寫進本快照
  - `[Phase 0]` Leanny 原始 key → 上述自有欄位的實際對應表,於抓取真實資料時建立(現階段不臆測 Leanny 的確切欄位名)。

### 4.3 圖像素材 — 預設一律自繪(避開高風險)
- **預設原則**:不使用任何從遊戲 dump 出的官方圖檔(武器圖、裝備圖等),那些是任天堂的美術著作;全部視覺以自繪 SVG 取代。
- 此原則為**預設部署與公開 repo 的合規底線**:正式對外的成品不得內含或散布任何官方美術。

#### 4.3.1 `[opt-in 例外]` 官方圖示 — 環境變數控制、執行時外部載入、不入庫
> 動機:開發 / 自架者可能想以官方圖示(主武器、副武器、特殊武器)輔助辨識。為在不破壞 §4.3 預設底線下提供此選項,採以下**嚴格界線**(任一條不滿足即不得開啟):
- **涵蓋範圍**:主武器圖示(來源 Leanny `weapon_flat`)、副 / 特殊武器圖示(來源 Leanny `subspe`);三者共用同一開關。
- **預設關閉**:由環境變數 `NEXT_PUBLIC_WEAPON_ICONS`(`on` 才啟用)控制,**預設 / 未設即關閉**;關閉時 UI 與版面與「全自繪」狀態一致,不顯示任何官方圖。
- **不入庫(硬條款)**:**任何官方圖檔(PNG 等)一律不得 commit 進 repo**。repo 內僅允許保留**檔名 / URL 參照字串**(metadata,如各筆 `iconName`),那是指標、非美術著作本體。
- **執行時外部載入**:啟用時才於執行時自外部來源 hotlink;base URL 可換:主武器 `NEXT_PUBLIC_WEAPON_ICON_BASE_URL`(預設 `weapon_flat`)、副/特殊 `NEXT_PUBLIC_WEAPON_ICON_SUBSPE_BASE_URL`(預設 `subspe`)。
- **風險歸屬**:開啟此 flag = 在自己的部署散布官方美術,合規風險由**開啟者自負**;專案預設部署不承擔此風險。
- 來源與數值同源(Leanny,§4.2 事實性數據立場);仍須維持 §4.4 非官方聲明與來源標註。

### 4.4 命名與品牌
- 站名與 UI 不得讓人誤以為是官方或任天堂公認的工具。
- 明確標示「非官方,與任天堂無關」。

### 4.5 資料版本與更新
- 武器數值會隨遊戲改版(平衡調整)變動;需指定一個快照版本,改版後再更新快照。

> `[已定案]` 首次快照採用**最新正式版**;確切版本號於 Phase 0 抓取資料時記錄並寫入快照(數值表需標註其對應的遊戲版本)。

## 5. 技術規格

### 5.1 技術棧(已定案)
- **框架**:React / Next.js(選此而非最熟的 Nuxt,目的是面試練習與展示技術廣度)。
- 建議使用 App Router,藉此練習 RSC / Server Components 等面試常考主題。

### 5.2 架構要點(建議)
- **資料策略**:武器數值為靜態資料,適合用 SSG / ISR(build 時或定期重新產生),藉此展示 Next.js 的資料取得策略。
  - `[補述 2026-06]` 渲染採「**server 殼 + client-only 島**」分工:首頁與**武器詳情頁 `/weapons/[id]` 維持 SSG**(數值靜態、利於 SEO);**瀏覽武器列表與隨機決定器兩頁改為 CSR 島** — 頁面殼與依 locale 解析的資料 props 仍由 server 預渲染,互動元件(WeaponList / RandomPicker)以 `next/dynamic({ ssr:false })` 掛載 + 固定高度骨架。此為對「全 SSG」的**刻意、局部偏離**,動機見下方「篩選條件持久化」補述。
- **多語系 (i18n)**:App Router 的 i18n routing 搭配 next-intl 之類方案。
- **樣式**:mobile-first,建議 Tailwind CSS。
- **SVG**:以 inline React component 實作,動態用 CSS / 輕量動畫函式庫。
- **隨機決定器**:client component,篩選條件以本地 state 管理。
- **篩選條件持久化(列表 + 隨機器)** `[補述 2026-06]`:兩處的篩選(分類 / 副 / 特殊 / 射程;隨機器另含多槽設定與「單次內去重」開關)以 **localStorage 暫存**,重新整理後沿用上次條件;還原時對當前快照清洗(剔除已不存在的 id、射程夾回邊界)。個人化偏好僅存在 client(不進伺服器、不依賴 cookie)。為消除 SSG 首屏「先畫預設 → 還原」的閃動,承載這兩處篩選的元件改為 **client-only 島**(見上方資料策略補述),首次 render 即讀 localStorage 直接落在還原態。抽選結果本身仍為瞬時、不持久化(維持 §3.2 無狀態純抽選)。
- **後端**:本專案資料為靜態 + 前端互動,原則上不需獨立後端。

### 5.3 部署
- 建議 Vercel(與 Next.js 整合最順)。
- (備選)Fly.io — 你已熟悉,可沿用既有 CI/CD 經驗。

### 5.4 非功能性需求
- **效能**:延續你既有的 Lighthouse CI 習慣,設定效能門檻。
- **無障礙 (a11y)**:語意化標籤、鍵盤可操作、對比度。
- **i18n**:三語切換,字串不寫死。
- **快取**:對 splatoon3.ink 資料做快取,遵守其抓取頻率要求。

## 6. MVP 與里程碑(建議分階段)

- **Phase 0 — 資料管線**:以最新正式版為快照基準並記錄版本號,依 §4.2 自建 schema、建立 Leanny key → 自有欄位對應表,產出武器資料快照 JSON(精簡核心子集數值 + 以 id 對應 splatoon3.ink 名稱)。
- **Phase 1 — 核心查詢**:武器列表 + 詳情(靜態表格),i18n,mobile-first。
- **Phase 2 — 隨機決定器**:多重條件篩選 + 抽選。
- **Phase 3 — SVG 美術**:11 分類 SVG + 最低限度動態,整合進詳情頁與抽選結果。
- **Phase 4 — 打磨上線**:效能、無障礙、部署、來源標註與非官方聲明。

## 8. 風險與注意事項摘要

- 整個 Splatoon 周邊工具生態多處於任天堂「默許」狀態,而非明文授權;「零風險」做不到,但本專案的設計(非營利、不用官方素材、不碰非公開 API、標明非官方)已將風險壓到最低。
- 開工前務必再次確認 Leanny 資料的實際授權狀態與 splatoon3.ink 當下的使用條款(條款可能隨時變動)。
